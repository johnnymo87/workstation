## Risk and rollback

- **Caching patch port introduces a regression**: most likely failure mode.
  Mitigation: `bun test` on the patched checkout BEFORE pushing the refresh
  commit catches the obvious cases. Rollback: `git revert` the rebase
  commit on opencode-cached, re-dispatch the build for v1.14.25 (which
  still works), accept that v1.14.28 isn't shipping yet.
- **opencode-patched build fails at the prefill-fix step despite local smoke
  test passing**: most likely cause is environmental drift (different bun
  version in CI vs local). Inspect the workflow log; refresh the patch if
  needed.
- **Workstation auto-PR fails to compute hashes**: 4 separate
  `nix-prefetch-url` calls; if one fails, the workflow exits non-zero and
  you do step 5 by hand using the inline sed commands as a template.
- **post-bump opencode crashes on startup**: roll back via `home-manager
  generations | head` then `nix-env --switch-generation N`.

## Why this plan exists (for future Claude)

The parent session (Plan 2 in `.plans/2026-04-27-task2-prefill-fix.md`) wrote
and pushed the prefill fix patch but stopped before the build/install/verify
chain because the user's `opencode-patched` build pipeline turned out to need
a lot more orchestration than a simple "push triggers build" assumption. This
plan captures the full orchestration so a future Claude doesn't have to
rediscover:

- That the build is `workflow_dispatch` only (not push-triggered).
- That `opencode-patched` consumes `caching.patch` from `opencode-cached`'s
  latest release, so opencode-cached must publish first.
- That both forks are typically refreshed together when upstream cuts a new
  version, but that's manual work.
- That the workstation auto-update flow keys on the version string and
  works cleanly when the upstream version actually changes (so
  same-version republishes are a degenerate case to be avoided).

The SPLIT execution (which immediately follows this plan in the parent
session) refreshes ONLY the `prefill-fix.patch` to v1.14.28 — the cheap
half. This plan documents the harder caching-patch port for execution in a
future session, when the user has the time and headspace.
