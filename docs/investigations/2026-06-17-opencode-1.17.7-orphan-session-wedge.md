## Q1 — Why sessions were created without a `session` row

### What the data shows (frozen backup + serve log)

- **22 true live orphans** (from `~/orphan-fix-sessions-20260617.txt`): each has **0**
  rows in `session`, `event_sequence`, `event`, `message`, `part`. Confirmed by direct
  `mode=ro` query.
- **All 22 were created by one process** — serve run `579b7ac6` (version **1.17.7**),
  interspersed **15:31→00:33 UTC** with **77 sibling creates in the same run that
  persisted fine** (104 creates total, ~26% lost). The `session` table's newest
  `time_created` froze at **00:35:50 UTC**, but `message`/`part` writes for *already-rowed*
  sessions kept landing until **03:11 / 03:37 UTC** — i.e. the write path stayed healthy;
  only these specific creates didn't persist.
- **No create-time error of any kind.** The *only* DB errors in the entire 115 MB log are
  **6 `FOREIGN KEY constraint failed`** Die's, all on **`part`/`message`** inserts via the
  `promptAsync` route (`handlers/session.ts:309-318`), 5 of them on orphan sessions. Zero
  `SessionAlreadyProjected`, `InvalidSyncEvent`, sequence/aggregate mismatch, `database is
  locked`, `SQLITE_BUSY`, disk-I/O, or UNIQUE errors anywhere.
- **Nothing deleted them.** There are **zero `session.deleted`/`session.removed` events**
  in the whole log (even the 5 orphans that got an agent `curl -X DELETE /session/…` —
  those DELETEs no-op'd on already-rowless sessions). `events.remove()` (the only path
  that wipes `event`/`event_sequence`) is called **only** from `session.remove()`
  (`session/session.ts:665`), which first emits a `Deleted` event — never seen.

### Code trace (deployed v1.17.7)

The create path is event-sourced and **atomic**:

- `Session.createNext` (`packages/opencode/src/session/session.ts:541-580`) builds an
  in-memory `Info`, logs `"created"`, then `yield* events.publish(SessionV1.Event.Created…)`
  and **`return result`** — it returns the *in-memory* object and never reads the persisted
  row back.
- The bridge (`packages/opencode/src/event-v2-bridge.ts:22-36`) forwards to core
  `EventV2.publish`; it does not swallow.
- `commitSyncEvent` (`packages/core/src/event.ts:215-383`) runs the **session-row INSERT
  (projector `packages/core/src/session/projector.ts:217-235`), the `event_sequence` INSERT,
  and the `event` INSERT inside one `db.transaction({behavior:"immediate"})`** (lines
  257-368). So a session row cannot exist without its events, and a *committed* `Created`
  cannot leave zero rows.
- The serve is **one process with one shared `Database` connection, one `EventV2` service,
  and a single 1-permit write semaphore** — `AppLayer`/`ManagedRuntime` are process-global
  and memoized (`packages/opencode/src/effect/app-runtime.ts:56-109`); the semaphore is
  `packages/core/src/database/sqlite.bun.ts:121`; instances are in-process *context* objects
  (`instance-store.ts`, no child processes). The sync registry that decides durability is a
  **module-global `Map` populated once at import** (`event.ts:85-133`); `session.created`
  has exactly one definition, and it carries `sync` (`packages/core/src/v1/session.ts:504-580`).

Consequences of that trace:

1. Because the registry is global + stable post-startup, `session.created` is durable for
   the entire process lifetime — **the "non-durable publish" path (`committed` falsy →
   `notify(false)`, no rows, no error) is impossible**, so it cannot explain an
   *intermittent* loss within one process.
2. A fresh `createNext` publish can therefore only **(a) commit all three rows** or
   **(b) die**. Evidence rules out (b): no error logged; the bus emitted `session.created`
   (the `notify()` that fires it is only reachable *after* a successful commit on this path);
   pigeon `registerSession ok=true`; `opencode-launch` got an id and proceeded to
   `prompt_async`. So the rows **were committed.**
3. They are nonetheless absent, and were **never deleted** (point above). The only path that
   removes `event` rows requires a `Deleted` event that never fired.

### Root cause

The committed `Created` transactions for these sessions **did not survive into the canonical
`opencode.db`**. The environment supplies the mechanism: the box ran the documented
**multi-writer topology** — the serve plus many standalone `opencode` processes all holding
the *single* `opencode.db` open — **across a binary cutover**. The log shows a real
**1.17.4 ↔ 1.17.7 version skew** creating sessions in the same window and **~54 concurrent
`run=` ids** during the incident. Concurrent multi-process writes to one SQLite-WAL file
during the cutover churn (binary swap + restarts while clients stayed attached) is the
root-cause class — exactly the split-brain the cutover runbook and `home.base.nix:306-309`
warn against ("a switch that swaps the opencode binary should stop ALL opencode processes at
once… Doing the switch from inside an opencode session will kill that session mid-switch").
gemini's 03:05 raw restart with 61 TUIs attached is the same class of move.

**The orphan phenomenon is an operational (topology) failure, not a 1.17.7 code regression.**
But two 1.17.x design facts turned a lost write into a hard incident, and are the real
fixable surface:

- **Event-sourcing FK coupling (1.16/1.17).** `part.message_id→message.id` and
  `message.session_id→session.id` mean a missing `session` row makes *every* later
  message/part projection fail FK — the "strange errors." The 1.15 line had no such
  event-sourced FK chain and degraded more gracefully.
- **`createNext` reports success on the in-memory object** (`session.ts:577-579`) and the
  session goes live (bus + pigeon registration) *before/independent of* a durable-write
  confirmation — unlike core `V2Session.create` (`packages/core/src/session.ts:200-253`),
  which reads the row back via `result.get(sessionID)` and handles `SessionAlreadyProjected`.
  So a lost write yields a **live-but-rowless** session instead of a clean failure.

**Residual uncertainty (honest):** with no incident-time WAL preserved and the backup frozen
*after* the loss, the exact SQLite-level frame-loss step (which connection's WAL frames were
discarded, and by which checkpoint/file operation) could not be pinned. Every *within-serve*
explanation, however, is definitively excluded by the trace above.

### Lead chased and ruled out: channel-suffixed DB split (latent risk)

The incident-fixer probed `~/.local/share/opencode/**opencode-local.db**` at 00:49 UTC — a
plausible second split-brain vector, because `Database.path()` (`packages/core/src/database/database.ts:43-55`)
returns `opencode-${InstallationChannel}.db` for any channel **not** in `{latest,beta,prod}`
(unless `OPENCODE_DB`/`OPENCODE_DISABLE_CHANNEL_DB` is set — **neither is set on cloudbox**;
the serve unit at `hosts/cloudbox/configuration.nix:490-589` pins no DB path). `opencode-local.db`
**does exist** (proof a non-prod channel wrote a separate file once) but is **stale — mtime
2026-06-08, 176 KB, 1 session, none of the orphans** — so it was **not** this incident's vector;
the loss was on the shared `opencode.db`. It is, however, a real **latent** split-brain risk: any
opencode invocation that resolves a different `InstallationChannel` (e.g. a from-source `bun run`,
a dev build) silently writes to a different DB file. **Hardening:** pin `OPENCODE_DB=opencode.db`
(or `OPENCODE_DISABLE_CHANNEL_DB=1`) in the shared environment so every invocation — serve, TUIs,
`opencode-launch` workers — targets the same file.

### Remediations (Q1)

- **Operational (highest leverage, no code):** never run two opencode binaries against
  `opencode.db` simultaneously. Switches/redeploys must `systemctl stop opencode-serve` +
  `pkill -f '\.opencode-wrapped'` from a bare SSH shell first (cutover runbook Phase 3). This
  alone prevents recurrence.
- **Robustness fix (code, upstream/fork):** make `createNext`
  (`packages/opencode/src/session/session.ts:541-580`) verify the durable write before
  returning — read the row back (mirror `packages/core/src/session.ts:251-253`) and fail
  loudly if absent, so a session can never go "live but rowless." File upstream; carry as a
  fork patch only if upstream is slow.
- **Detection:** a tiny periodic invariant check (sessions live in serve memory / registered
  with pigeon but absent from `session`) would have surfaced this in minutes.

## Q2 — The serve wedge under thundering-herd reconnect (regression: YES)

### What `project copy refresh` is and why it's O(n)

`ProjectCopy.refreshAfterBoot` (`packages/core/src/project/copy.ts:126-144`) is wired into
the **per-location layer** (`packages/core/src/location-layer.ts:106-123`, `Layer.fresh`,
`idleTimeToLive: "60 minutes"`), so it fires **once per location/instance boot** — including
every reconnect after the 60-min idle eviction. `refresh()` (`copy.ts:233-288`):

1. `directories.list(projectID)` — all tracked dirs (root + every worktree copy) for the project.
2. `fs.isDir` on **each**, `{ concurrency: "unbounded" }`.
3. For each source dir, run the git-worktree strategy `.list()` — a **`git worktree list`
   subprocess** — also `{ concurrency: "unbounded" }`.
4. One `db.transaction` writing `directories.create`/`remove` for **every** discovered/removed dir.

For a project like `mono` with dozens of worktrees this is real work, and it runs **per
reconnecting instance**.

### Evidence it caused the wedge

- Log: **169 `project copy refresh started` vs 141 `done` → 28 refreshes hung incomplete**
  (in-flight when the event loop blocked).
- **Bursts of 16-20 concurrent refreshes** in single minutes during the restart/reconnect
  window (e.g. 20 in `04:04`, 16 in `03:56` UTC) — the thundering herd.
- One refresh's `done` line listed up to **136 directories** touched (the O(n) factor).
- `677 "global event connected"` reconnect events — the herd.

Mechanism: a serve restart (or `reset-workspace`) → ~61 TUIs reconnect → ~61 location boots →
~61 `project copy refresh`es, each spawning unbounded `git worktree list` subprocesses + `fs`
stats and a write transaction (serialized through the single 1-permit DB semaphore). The git
subprocess fan-out + fs work pegs one core / blocks the Bun event loop; the herd of reconnect
event-streams (plus the 2M-token gemini session being re-read) drives RSS to ~19.6 GB. HTTP
goes unresponsive; the log freezes.

### Regression confirmation

`packages/core/src/project/copy.ts` and `project/directories.ts` are **ABSENT at
`v1.15.13`**; introduced **2026-06-02 via PR #30139** ("project copying and tracking
directories", migration `20260602182828_add_project_directories`). So `project copy refresh`
— and its reconnect-storm amplification — is **new on the 1.16/1.17 line** the box did not run
before. This is a genuine 1.17.x regression for our many-worktree, many-reconnect workload.

### Remediations (Q2)

- **Bound the fan-out:** replace `{ concurrency: "unbounded" }` (copy.ts:238, 257) with a
  small fixed concurrency, and/or **dedupe/debounce refresh per `projectID`** so 61 instance
  boots of the same project coalesce into one refresh (it already keys on
  `location.project.id`). Upstream candidate.
- **Pace reconnects:** keep using the paced `~/restore-work-sessions.sh` (5s gap + health
  gate); never re-open ~40 tabs at once. This is the practical mitigation today.
- **Cap session re-read cost** on reconnect for the 2M-token poison session (don't auto-open it).

### Q2 — SHIPPED (2026-06-23, bead workstation-sqd5)

The "bound the fan-out + dedupe per projectID" remediation landed as
`opencode-patched` patch **`project-copy-debounce.patch`** (commit `229d76e`),

## Q3 — `reset-workspace` snapshot deadlock (confirmed)

`pkgs/reset-workspace/default.nix:257`:

```sh
resolved_sid=$(curl -fsS --get "$OPENCODE_URL/session" \
  --data-urlencode "directory=$cwd" --data-urlencode "roots=true" --data-urlencode "limit=1" \
  2>/dev/null | jq -r '.[0].id // empty' 2>/dev/null || true)
```

This **manifest-capture** curl has **no `--max-time`/`--connect-timeout`**. It runs in **Step 1
(manifest capture), before Step 5 (the force-restart of `opencode-serve`)**. Against a wedged
serve (event loop blocked but the OS still completing the TCP handshake), curl connects and then
**blocks on read forever**; the `2>/dev/null || true` only catches error *exits*, not a hang. The
script never reaches the restart that would clear the wedge → **deadlock** (it hung tonight). The
team already learned this exact lesson for the health-check curls (lines 338-352:
`--max-time 3`, "curl parked for 6+ hours on 2026-05-16") but did not apply it to line 257.

### Remediations (Q3)

- **Minimal:** add `--max-time 5 --connect-timeout 3` to the line-257 curl (matches the
  existing pattern). A wedged serve then fails that one pid fast (skip-restore) and the loop
  continues to the restart.
- **Defense-in-depth (recommended):** make the force-restart effectively unconditional by
  probing health *before* manifest capture — if `curl -sf --max-time 3 …/global/health` fails,
  log it, **skip capture, and go straight to the restart**. The restart is the whole point of
  the tool; it should never be gated behind a call that depends on the (possibly wedged) serve.

## Q4 — The swallow patch + the two `origin/main` commits

### What the patch actually does (assessed, not relied on)

`patches/sqlite-foreign-key-wrap.patch` (in `opencode-patched@26b5ec6`) wraps the
`MessageUpdated` and `PartUpdated` projector inserts (`projector.ts:272-276` and `322-326`)
with `Effect.catch(foreign → Effect.void)` before `Effect.orDie`. Because the projector runs
**inside** `commitSyncEvent`'s transaction (`event.ts:336-338`), swallowing the FK error makes
the projector "succeed," so the transaction **continues and commits the `event_sequence`/`event`
rows while silently skipping the message/part projection**. Net effect on a recurrence: the
event log advances for an orphan session while its projections are dropped — *worse* than the
loud, transaction-rolling-back FK error it replaced, and it does **not** prevent any cascade
(projectors run inline per-publish; an unswallowed FK fails only that one session's write, not
others). It is currently **dormant** (the real cause — missing `session` rows — is fixed).

### The two `origin/main` commits

- `opencode-patched 26b5ec6` — adds the patch + `apply.sh`/`README`/`build-release.yml` entry.
- `workstation 5cb129c` — bumps the 4 `opencode-platforms` hashes in `home.base.nix` to the
  rebuilt `v1.17.7-patched` that includes the patch.

### Recommendation: **revert** (preferred), or change to **log-and-continue**

The patch fixes a symptom, not the cause (Q1), and provides no real protection. Prefer a clean
revert; if a belt-and-suspenders guard against a future orphan recurrence wedging a session is
wanted, keep the catch but `Effect.logWarning` (with `sessionID`) before `Effect.void` so it is
visible — and pair it with the `createNext` read-back fix so orphans can't go live at all.

## Cross-cutting takeaway

Q1 and Q2 share one root: **1.17.x is far less tolerant of this box's many-process,
many-worktree, frequent-reconnect topology than the 1.15 line was** — event-sourcing adds hard
FK coupling (Q1's lost-write becomes a hard failure) and per-boot `project copy refresh` adds
O(n) reconnect cost (Q2's wedge). The durable fixes are: (a) never multi-write the DB across a
binary swap, (b) make `createNext` verify its write, (c) debounce/bound `project copy refresh`,
and (d) give `reset-workspace` a timeout + unconditional restart so it can always recover the box.
