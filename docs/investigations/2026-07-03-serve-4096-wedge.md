# 2026-07-03 devbox serve-0 (:4096) wedge at the nightly reset — forensics

**Bead:** workstation-94g8 · **Host:** devbox · **Binary:** opencode-patched 1.17.7.9
(v1.17.7-patched.9, released 2026-06-28) · **Investigated:** 2026-07-03 (read-only)

## Verdict (one paragraph)

The main JS event loop of the 24h-old serve-0 process (pid 82581) froze hard at
~06:58–07:00 UTC (02:58–03:00 EDT) and stayed frozen for ≥93 s — long enough to
swallow both the reset's 3 s `/global/health` probe (07:00:06–09Z) and the
entire 90 s SIGTERM window (07:00:09→07:01:39Z, SIGTERM is a JS-level handler
and never ran) — forcing SIGKILL. This was an **acute** freeze, not a chronic
5-hour stall: the process fired its scheduled timers with ~125 ms/hr drift all
night, last at 06:58:22Z, 107 s before SIGTERM. The best-supported proximate
mechanism is a **memory-pressure stall at the cgroup MemoryHigh=4G ceiling**:
the invocation had been pinned at the 4G ceiling (journald: "4G memory peak,
434.8M memory swap peak") after an unusually heavy day dominated by giant
persistent eternal-machinery swarm sessions, and the hourly internally-driven
instance re-boot of the (large, 7-worktree) eternal-machinery location at
06:58:22Z — the process's last logged act — supplied an allocation burst that
drove the allocating main thread into sustained cgroup direct-reclaim/
`memory.high` throttling (plausibly compounded by a long full GC over a
partially-swapped heap). Structurally, the unit's protection scheme
(MemoryHigh=4G soft, MemoryMax=5G hard, Restart=always) **cannot recover from
this state**: `memory.high` reclaim keeps usage pinned at 4G so MemoryMax/OOM/
Restart never fire, and the worker-thread heartbeat (serve-lease Fix C) keeps
reporting the serve healthy to pigeon while the main loop is dead — a wedge
that is invisible to every existing liveness mechanism until something probes
HTTP.

is ruled OUT as the mechanism, but the boot still allocates substantially.

### 4. Memory state and the structural trap

- Unit limits (`users/dev/home.devbox.nix`): MemoryHigh=4G, MemoryMax=5G,
  OOMScoreAdjust=500, Restart=always, no TimeoutStopSec (default 90 s).
- Kernel journal Jul 2 20:00Z → Jul 3 07:05Z: **zero** OOM/memory events. The
  cgroup never hit MemoryMax=5G, so the only auto-recovery path
  (OOM-kill → Restart=always) never armed. `memory.high` semantics guarantee
  this: over-high usage is clamped by direct reclaim performed *by the
  allocating task* plus penalty throttling — the process gets slower, not
  dead. systemd-oomd is running but has no ManagedOOM* policy on this slice.
- Swap is zram (22.9G, fast), so pure swap-in latency alone likely cannot
  explain a 93 s freeze; the freeze needs the reclaim/throttle penalty path
  (allocating thread repeatedly parked in `mem_cgroup_handle_over_high`-style
  sleeps) and/or a full GC walking a partially-swapped multi-GB heap.
- The wedge invocation demonstrably visited the ceiling (4G peak, 435M swap
  peak). What is NOT provable retroactively: `memory.current` at 06:58Z
  (the cgroup died with the process; no PSI history is recorded). Note the
  serve's page-cache for the 4.4 GB shared `opencode.db` is charged to the
  serve's cgroup, which makes "parked at the ceiling" the steady state for a
  busy serve even after its JS heap quiets down.
- Fresh serves idle far below the ceiling (current 4096 after 4.5 h:
  memory.current 0.88G, peak 0.93G, swap 0, PSI all-zero) — the ceiling is a
  load/accumulation phenomenon, reached within hours under real work.

## Root cause statement

**Proximate (best-supported hypothesis):** After ~19 h of heavy load dominated
by multi-thousand-message eternal-machinery swarm sessions (plus the anchor
role), serve-0's cgroup was parked at the MemoryHigh=4G ceiling with ~0.4G
pushed to zram. The hourly idle-TTL re-boot of the eternal-machinery location
at 06:58:22Z — the process's last logged act — triggered an allocation burst
at the ceiling, throwing the allocating main thread into sustained
`memory.high` direct-reclaim/penalty throttling (likely compounded by full-GC
work over a partially swapped heap). The main event loop froze for ≥93 s,
exactly spanning the reset's health probe and SIGTERM window. Confidence:
moderate-high for "memory-pressure stall seeded by the 06:58 boot"; the
freeze-of-the-main-loop itself and its onset window (06:58:22–07:00:06Z) are
proven. An alternative acute mechanism (a hang inside the boot's layer build
unrelated to memory, e.g. watcher/plugin init) cannot be fully excluded with
the process dead — see "missing evidence".

**Structural (why it wedged instead of restarting):**
1. **MemoryHigh=4G with no responsiveness-based recovery.** `memory.high`
   reclaim clamps usage below MemoryMax forever, so the
   MemoryMax→OOM-kill→Restart=always path never fires. The failure mode is
   "alive but frozen", the one state nothing watches for.
2. **Fix C made the wedge invisible.** Moving the serve heartbeat to a worker
   thread (to stop false-positive lease flapping) also removed true-positive
   detection: pigeon saw a healthy serve all night. Liveness now attests "the
   worker thread can write sqlite", not "the serve can serve".
3. **Anchor concentration.** serve-0 takes all new sessions + fallback (K=2),
   so retained heap concentrates there (journald history table above).
4. **Unbounded session growth.** 7k-message sessions and a 4.4 GB shared DB
   push any serve that runs their turns toward the ceiling.
5. **No pre-kill forensics.** The reset goes probe-fail → SIGTERM → SIGKILL
   without capturing `/proc/<pid>/stack`, `/proc/<pid>/status`,
   `cgroup memory.stat`/PSI — so each wedge destroys its own evidence.

## Missing evidence to fully close

- No PSI/memory.current time series for the dead cgroup (only journald peaks).
- No thread stacks at freeze time (SIGKILL, no core).
- CPU user/sys split for the invocation (would confirm reclaim burn).
- Whether memory.current was still ~4G at 06:58Z after 5 h idle (page-cache
  charging makes it likely, not proven).

## Proposed fixes (NOT implemented — decision needed)

1. **Responsiveness-based liveness:** have pigeon's sweep (or a timer unit)
   probe each serve's `/global/health` with a short timeout and mark/restart on
   N consecutive failures — heartbeat_at alone is provably insufficient now.
2. **Pre-kill diagnostics in reset-workspace:** on probe failure, snapshot
   `/proc/<pid>/{stack,status,smaps_rollup}`, `memory.{current,stat,pressure}`
   before SIGTERM (cheap, read-only, closes the mechanism question on next
   occurrence).
3. **Rethink the memory envelope:** either narrow the MemoryHigh→MemoryMax gap
   (e.g. 4G→4.25G) or drop MemoryHigh entirely on the serve units so a
   ballooned serve OOM-kills and Restart=always recovers it in seconds
   (sessions survive in the DB); or add systemd-oomd
   `ManagedOOMMemoryPressure=kill` for the serve slice.
4. **Bound the workload:** rotate/compact the eternal-machinery mega-sessions
   (7k messages) and consider a session-size alarm; they are the dominant heap
   driver on both serves.
5. **(Nice-to-have)** proactive restart when memory.current > threshold for
   M minutes (the reset skill already suggests manual resets "past ~6 GB" — on
   devbox the per-instance ceiling is 4G, so that guidance never triggers).
