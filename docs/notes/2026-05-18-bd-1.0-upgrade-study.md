# beads 0.x → 1.0 upgrade study

Date: 2026-05-18
Author: opencode research session
Trigger: commit `00befba` bumped beads 0.49.1 → 1.0.4 in `pkgs/beads/default.nix`.
First runs of `bd 1.0.4` against our existing `.beads/` created new directories
(`embeddeddolt/`, `backup/`, `export-state.json`), broke `bd doctor`, and revealed
that `bd sync` (the subcommand) no longer exists.

This is a research-only writeup. No code changes were made. Recommendations are
deferred for review.

## TL;DR

0. **We are already losing data.** Comparing `git show HEAD:.beads/issues.jsonl`
   (132 lines) against the current working-tree `.beads/issues.jsonl` (101 lines)
   shows 31 issue records gone, including in-progress and open beads
   (workstation-0de "self-compact regression on 1.15.0", workstation-0qg
   "system-prompt-split.patch", workstation-1lp "oc-auto-attach pigeon-daemon
   silent drop", workstation-70i, workstation-7kn, workstation-aao,
   workstation-adw, workstation-b4p, …). `bd show workstation-0qg` and `bd show
   workstation-0de` both return *"no issue found"* — the live Dolt DB doesn't
   have them either. This matches the **#3948 footprint** (auto-import on
   bd startup walks JSONL into Dolt; if it ran when the DB was empty or
   metadata-drifted, it would re-import only what was in the JSONL at that
   moment). The current state: HEAD's `.beads/issues.jsonl` is the only
   surviving copy. **Do NOT `git checkout` or `git restore` `.beads/issues.jsonl`
   to "clean up" the working tree — that would commit the loss.** Recovery
   is a separate, careful workstream: roll back the working-tree JSONL to
   match HEAD, set `import.auto=false` to prevent the next bd invocation from
   re-dropping it, then `bd import .beads/issues.jsonl` deliberately into the
   live Dolt DB. See R0 below.

1. **The fork situation is a clean transfer, not a divergence.** `steveyegge/beads`
   was renamed to `gastownhall/beads`. GitHub serves a 301 redirect; the two
   "repos" resolve to the same `repo_id`. Our `fetchFromGitHub { owner = "steveyegge"; ... }`
   still works via the redirect, but the canonical owner is `gastownhall`.

2. **`bd sync` was removed in v0.56.0** (Feb 2026), not in 1.0. We've been
   broken for months and didn't notice because we never actually hit the
   pre-commit hook on a tracked file (or because the daemon's 30s debounce was
   doing the work). Replacement: `bd dolt push` / `bd dolt pull` for cross-machine
   sync, plus `bd export` / `bd import` for JSONL interchange. The
   `--flush-only` flow no longer exists — flushes are implicit on every write
   under `dolt.auto-commit=on` (the default).

3. **Embedded Dolt is the default on every platform as of 1.0.0**, including
   macOS via CGO. `.beads/embeddeddolt/<prefix>/` is the live database; the old
   SQLite `beads.db*` files lying around our workspace are dead weight. v1.0.4
   in particular removed the process-lifetime flock around embedded mode (PR
   #3614), so multiple concurrent `bd` calls now open the embedded engine
   independently and rely on Dolt's own concurrency for safety.

4. **Our `--skip-hooks` wrapper in `pkgs/beads/default.nix` is still
   correct** mechanically (the flag still exists), but the *rationale* in the
   comment is stale: it references `bd sync --flush-only` and the SQLite-era
   "inline" hook variant. The new hook is a thin shim that delegates to
   `bd hooks run pre-commit`, and `bd init --non-interactive` no longer
   installs git hooks at all on a fresh init — they only land via an explicit
   `bd hooks install` or `bd doctor --fix`. So the wrapper is now mostly
   defensive belt-and-suspenders.

5. **Two open data-loss regressions in 1.0.4 affect us.** Both are filed
   against gastownhall/beads:
   - **#4033**: in embedded mode, if the Dolt DB is empty but `.beads/issues.jsonl`
     is populated, the auto-export silently *deletes* the JSONL. No warning, no
     daemon involvement.
   - **#3948**: `maybeAutoImportJSONL`'s `TotalIssues > 0` no-op guard fails to
     fire under some conditions, causing every `bd` invocation to re-import the
     JSONL (`INSERT … ON DUPLICATE KEY UPDATE`) over the live Dolt state,
     silently reverting recent writes.

   Both have `export.auto=true` and `import.auto=true` as the default, so we
   are exposed unless we explicitly disable one or the other.

## 2. The 0.x → 1.0 release notes

Read using `gh release view vX.Y.Z --repo gastownhall/beads` and
`/tmp/beads-1.0/CHANGELOG.md`. The CHANGELOG is the canonical source — it is
hand-written and well-curated. The GitHub release "Others" sections are
autogenerated from commit subjects and are noisy.

### v1.0.0 (2026-04-02) — the headline release

The CHANGELOG framing is:
> Beads 1.0 marks the transition from rapid iteration to production stability.
> The Dolt migration that began in v0.55.0 is complete: embedded Dolt is the
> default on all platforms, server lifecycle management is no longer required,
> and the critical reliability issues that affected the v0.55–v0.63 series have
> been resolved.

Key additions:
- **Embedded Dolt default on macOS** via native CGO builds (PR #2971). Removes
  the need to manage an external `dolt sql-server`.
- **Custom statuses/types** moved into normalized SQL tables instead of
  comma-separated config strings.
- **First-class issue types**: `spike`, `story`, `milestone` joined `task`/`bug`/
  `feature`/`epic`.

Changed:
- `bd setup claude` defaults to project-local (writes to `.claude/settings.json`)
  and **auto-runs during `bd init`** — this is why our fresh `bd init` in a
  disposable workspace created `.claude/` and a `CLAUDE.md`.

### v1.0.1 (2026-04-15) — auto-export turned on by default

> **Auto-export enabled by default** — `export.auto` defaults to `true` and the
> default `export.path` is now `issues.jsonl` (previously `export.jsonl`), with
> `export.git-add` on by default.

This is the biggest behavior change for us. v1.0.4's defaults in a fresh
workspace:

```
export.auto       = true     (default)
export.git-add    = true     (default)
export.interval   = 60s      (default)
export.path       = issues.jsonl (default)
import.auto       = true     (default)
dolt.auto-commit  = on       (default)
backup.enabled    = false    (default)
backup.git-push   = false    (default)
```

Also new in 1.0.1:
- `bd batch` — atomic multi-op transactions
- `bd config show / drift / apply` — config provenance + drift reconciliation
- `BEADS_DOLT_READY_TIMEOUT` env to tune the 10s `waitForReady` deadline

### v1.0.2 (2026-04-15) — npm provenance hotfix only

One-line release: fixed `npm-package/package.json` URLs to satisfy sigstore
provenance after the repo move.

### v1.0.3 (2026-04-24)

- **`go install ...@latest` restored** by removing local `replace` directives
  from `go.mod`. (Doesn't matter to us — we use `buildGoModule`/`buildGo126Module`.)
- Embedded pre-commit hook no longer prints "no beads database found".
- Some Windows/macOS portability fixes.

### v1.0.4 (2026-05-07) — concurrency model change + init safety

CHANGELOG highlights:

> **`beads.OpenBestAvailable` signature changed (breaking)** — returns
> `(Storage, error)` instead of `(Storage, Unlocker, error)`. The per-open
> flock and `Unlocker` return value were removed; the embedded Dolt engine
> handles its own concurrency internally. External SDK consumers that call
> `OpenBestAvailable` must drop the second return value. (PR #3614)
>
> **Embedded-mode flock removed** — the process-lifetime exclusive flock on
> `.beads/embeddeddolt/` has been removed. Concurrent `bd` processes now open
> the embedded engine independently. If the GH#2571 nil-deref stack
> (`NewConnector → DoltDB.SetCrashOnFatalError → CollectDBs`) resurfaces under
> concurrent access, it should be filed and fixed in `dolthub/driver`, not
> reintroduced as a beads-side flock. (PR #3614)

This is relevant because it means multiple concurrent `bd` calls (e.g. from
parallel opencode sessions, or our daemonless workflow) no longer serialize on
a flock. If we hit the original race (#2571), the symptom will be a nil-deref
panic in dolthub's driver, not an "embedded store is locked" error.

Also:
- New `bd init --reinit-local` / `--discard-remote` flags replace the
  overloaded `--force`. Old `--force` still works as an alias but emits a
  deprecation warning.
- `bd -C <path>` (like `git -C`).
- Several `fix(hooks): …` entries (preserve user content during migration,
  use 0700 perms, etc.) — these explain why our `bd init` reshapes hooks.

### Removed/breaking surface that affects us

| Surface | Status | Replacement | Removed in |
|---------|--------|-------------|-----------|
| `bd sync` subcommand | deprecated no-op since v0.56.0 | `bd dolt push` / `bd dolt pull` + automatic `dolt.auto-commit=on` flush | v0.56.0 (Feb 2026) |
| `bd sync --flush-only` | gone with `bd sync` | implicit on every write under default config | v0.56.0 |
| `bd sync --check` / `--from-main` / `--squash` flags | gone | n/a | v0.56.0 |
| `git-portable mode` / `belt-and-suspenders mode` | removed | Dolt-native push/pull only | v0.56.0 |
| JSONL bootstrap path on clone | removed | `bd bootstrap` (Dolt clone only) | v0.56.0 |
| Embedded Dolt mode | **un**-removed in v1.0.0 (it had been removed in v0.56.0!) | now the default again | v0.56.0 (removed) / v1.0.0 (re-added as default) |
| `bd init --force` (overloaded) | deprecated alias | `--reinit-local` (local-only) + `--discard-remote` (remote override) | v1.0.4 (still works, warns) |
| `beads.OpenBestAvailable` Go API: `(Storage, Unlocker, error)` | breaking signature change | `(Storage, error)` | v1.0.4 |

Two things worth re-emphasizing:

- Embedded mode was *removed* in v0.56.0 (Feb 2026) and *brought back* as the
  default in v1.0.0 (Apr 2026). Anyone who read documentation from the
  v0.56–v0.63 era will see a different story than what 1.0 actually ships.
- We jumped 0.49.1 → 1.0.4. That straddles both the removal *and* the
  re-introduction of embedded mode, plus the `bd sync` removal. Our workspace
  is doing the multi-step migration on its own.

## 4. Our pre-commit hook situation

### What the wrapper in `pkgs/beads/default.nix` does today

```sh
if [ "$1" = "init" ]; then
  # ... if user didn't already pass --skip-hooks ...
  exec $BD_REAL init --skip-hooks "$@"
fi
exec $BD_REAL "$@"
```

The `--skip-hooks` flag **still exists** in `bd 1.0.4`:

```
$ bd init --help | grep -A1 skip-hooks
      --skip-hooks               Skip git hooks installation
```

So the wrapper continues to suppress `bd init`-time hook installation. Verified
by `which bd` resolving to our wrapper and `cat $(which bd)` showing the
expected shim.

### What's stale in the comment

The justification comment (`pkgs/beads/default.nix:32–60`) says:

> Why: upstream `bd init` … installs git hooks by default into .git/hooks/
> for SQLite-backed repos. The pre-commit hook it writes … is the inline
> variant which has a worktree bug: it correctly resolves the main repo's
> .beads directory into a `BEADS_DIR` shell variable, but never exports it
> nor cd's to the main repo before running `bd sync --flush-only`. … the bd
> daemon already auto-flushes on a 30s debounce and we run `bd sync` manually
> at session end.

Every concrete claim in this paragraph is now wrong:

1. **SQLite backend is gone.** Embedded Dolt is the default; the SQLite ↔
   Dolt bifurcation was removed in v0.56.0 and Dolt embedded was made
   default again in v1.0.0.
2. **The inline hook variant is gone.** v1.0.4 hooks are *thin shims* that
   delegate to `bd hooks run pre-commit`. Reproduced in a disposable
   workspace:

   ```sh
   $ bd hooks install
   $ cat .git/hooks/pre-commit
   #!/usr/bin/env sh
   # --- BEGIN BEADS INTEGRATION v1.0.4 ---
   if command -v bd >/dev/null 2>&1; then
     export BD_GIT_HOOK=1
     _bd_timeout=${BEADS_HOOK_TIMEOUT:-300}
     if command -v timeout >/dev/null 2>&1; then
       timeout "$_bd_timeout" bd hooks run pre-commit "$@"
       ...
     fi
     ...
   fi
   # --- END BEADS INTEGRATION v1.0.4 ---
   ```

   No `BEADS_DIR`, no `bd sync --flush-only`. The actual flush/export logic
   moved into the bd binary and runs under `bd hooks run pre-commit`. Whether
   that has its own worktree bug is a separate question.
3. **`bd sync --flush-only` is gone.** `bd sync` was removed in v0.56.0
   (Feb 2026); the daemon-flush-on-30s-debounce concept is also gone (no
   daemon under embedded mode). Flushes happen synchronously on every write
   under `dolt.auto-commit=on` (the default).
4. **"We run `bd sync` manually at session end"** — `bd sync` is a no-op.
   The repo-level `AGENTS.md` still has it in the "Landing the plane"
   workflow (see "MANDATORY WORKFLOW" section). That `bd sync` line is
   harmless (no-op) but misleading. The actual auto-flush is `dolt.auto-commit=on`
   and the cross-machine sync is `bd dolt push` (when configured).

### Does `bd init` install a hook by default in 1.0.4?

**No** — `bd init --non-interactive` in a disposable workspace installed
*Claude* hooks (`.claude/settings.json` for SessionStart and PreCompact) but
left `.git/hooks/pre-commit` absent. Git hooks are only installed when:

- the user passes interactive `bd init` and answers yes to a prompt, or
- the user runs `bd hooks install` explicitly, or
- `bd doctor --fix` is run (and detects the missing hooks).

So our `--skip-hooks` wrapper is now belt-and-suspenders. It's still a
correct policy lever — it just no longer prevents the noisy default it was
written to prevent.

### Was a stale hook left on this machine?

The user mentions a deleted `.git/hooks/pre-commit`. Current state on this
checkout:

```
$ ls -la .git/hooks/ | grep -v sample
total 128
drwxr-xr-x  16 ...  .
drwxr-xr-x  17 ...  ..
```

No active hooks. We may want to audit cloudbox / devbox / crostini for stale
0.x-shaped pre-commit hooks that still call `bd sync --flush-only`. Those
would silently no-op on 1.0 (since `bd sync` is gone), but they'd add
unnecessary `bd` invocations to every commit.

## 5. The `embeddeddolt` backend

### Where it's at in 1.0.4

- **Embedded mode is the default** for fresh `bd init` (verified in a
  disposable workspace).
- The store lives at `.beads/embeddeddolt/<dolt_database>/` where
  `<dolt_database>` is derived from the issue prefix (`bd-experiment` →
  `bd_experiment`).
- There is no separate Dolt SQL server process. The Dolt engine runs in-process.
- v1.0.4 removed the process-lifetime flock around the embedded store; the
  Dolt engine handles concurrency internally (PR #3614).
- `bd doctor` does NOT support embedded mode yet (#3597, #3794). It exits with
  a "not yet supported" message.

### Production-readiness

The CHANGELOG claims production stability with v1.0.0:
> Beads 1.0 marks the transition from rapid iteration to production stability.

But the open-issue tracker tells a more nuanced story:

- The flock-removal in v1.0.4 (~10 days ago) was a substantive concurrency
  change — the old behavior had been in place for the GH#2571 race fix, and
  upstream is explicit that if the race resurfaces, the fix needs to be in
  dolthub's driver.
- Two distinct silent data-loss bugs (#4033, #3948) are open against 1.0.4
  *as of this week*. Both involve the auto-export/auto-import surface that
  embedded mode enables by default.
- `bd doctor` cannot diagnose embedded-mode workspaces yet (#3597, #3794).

Verdict: **functional and increasingly default, but not yet quiet.** The
upstream cadence is brisk (lots of PRs each day), but there are unresolved
data-integrity issues on the default code path.

So bd 1.0.4 **silently migrated** us — the live data is in `.beads/embeddeddolt/`,

which means bd is falling through some compatibility path to infer the mode.
That's exactly the kind of state that the open-issue #4033 / #3948 surface
("metadata.json drift triggers data-loss path") was filed for. We are
adjacent to that footgun.

### Auto-migration story

The CHANGELOG references the SQLite → Dolt migration heavily across v0.55–v0.63:

> **Auto-migration shim** — SQLite to Dolt migration via sqlite3 CLI
> **Auto-migrate on first command** — SQLite to Dolt auto-migration on first
> `bd` command

So the auto-migration *did* fire when we first ran 1.0.4 — that's why
`embeddeddolt/` materialized. But the leftover `beads.db*` files were never
cleaned up, and the leftover `daemon.*` and `bd.sock` files predate any
auto-cleanup the migration would do.

### Multi-machine sync failure modes

The scenario the prompt asks about: "what if multiple machines bd-sync (git
push/pull of `.beads/issues.jsonl`) and one is on embedded mode and another
isn't?"

The honest answer from reading the docs:

- We are **not currently doing Dolt-native push/pull** (no `bd dolt remote
  add origin …` step has been run). Our workspace has a `.beads/issues.jsonl`
  committed to git, and that's it. No `refs/dolt/data` on `origin`.
- 1.0's design is explicit: `.beads/issues.jsonl` is an *export*, not a
  sync channel. Routine `bd import .beads/issues.jsonl` is upsert-only and
  cannot infer deletions. ([SYNC_CONCEPTS.md])
- Each machine's Dolt DB therefore evolves independently. As long as
  `export.auto=true` is on, each machine's pre-commit will re-export its
  own DB state into the committed JSONL. Git merges in `issues.jsonl` will
  be ad-hoc text merges. Re-imports across machines will be upserts (no
  deletions propagate).
- Mode-mismatch (one embedded, one server) at the *bd backend level* is
  invisible at the JSONL layer — JSONL is identical between modes.

So the current failure modes are:

1. **Lost deletes.** If A deletes an issue, B will re-import it on the next
   `git pull` (`import.auto=true`).
2. **Conflicting writes.** If A and B both touch the same issue, the next
   round of import-then-export will pick whichever JSONL line landed in
   `issues.jsonl` after the git merge. Cell-level merge (Dolt's claim to
   fame) is not in play here at all.
3. **#4033 hazard.** If any machine ever ends up with an empty Dolt DB
   (post-init, post-`--reinit-local`, post-disk-loss), the auto-export
   silently deletes the committed JSONL — and then the next `git commit`
   stages that deletion.

The "right" 1.0 sync model is `bd dolt push` / `bd dolt pull` against a
shared Dolt remote (Dolt-on-Git via `refs/dolt/data`, DoltHub, S3, GCS).
See `docs/SYNC_CONCEPTS.md` and `docs/SYNC_SETUP.md` in the beads repo.
Adopting that would also need infra: every host that ever calls `bd dolt
push` needs the dolt binary itself, 1.88.1+ (per `docs/SYNC_SETUP.md`).
