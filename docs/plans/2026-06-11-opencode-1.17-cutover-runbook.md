# OpenCode 1.15 → 1.17 Cutover Runbook

**Status:** ready to execute (validated 2026-06-11). Supersedes the indefinite
hold described in the `holding-opencode-on-1.15` skill — the DB-corruption fears
that motivated the hold were empirically disproven for our topology on v1.17.2.

**Goal:** move every opencode entry point from `v1.15.13-patched.3` to a
`1.17.x-patched` fork release, on cloudbox (and devbox), with no DB corruption.

## Why this is safe now (evidence, 2026-06-11)

All tested against real upstream **v1.17.2** run from source, isolated under
`/tmp` (the live 4.2 GB DB was never touched). See the `holding-opencode-on-1.15`
skill for the full writeup.

- **#31072 (subagent seq race) does NOT reproduce on our topology.** 42 subagent
  sessions across in-process (10 parallel), multi-process (13-way), and combined
  worst-case (4 procs × 8 parallel) stress → **0 orphaned, 0 missing
  `session_message`, 0 duplicate `(aggregate_id,seq)`, `quick_check ok`.** Root
  cause it can't bite: `commitSyncEvent` reads `latest` *inside* a `BEGIN
  IMMEDIATE` transaction, and intra-process a 1-permit semaphore
  (`sqlite.bun.ts`) serializes all transactions. The reporter's 99.4% was the
  multi-process desktop app on an older snapshot.
- **Old history loads fine (#29908 does not bite us).** 64/64 sampled sessions
  (6 oldest + 50 random + newest) loaded; 142,974 messages / 591,545 parts intact.
- **The destructive migration won't re-fire.** Our DB already records all 32
  migrations (incl. `20260604172448`) as applied, so 1.17 re-runs nothing on boot.
- **The one real gotcha for the migrate path:** our DB is a *repaired hybrid* —
  `fix.sh` dropped `session_message.seq` while its migration is marked done, so
  1.17 can't write (`table session_message has no column named seq` at
  `prompt.ts:1110`) until we replay that one step. Fix validated below.

## Topology reality (read before quiescing)

cloudbox is **multi-writer**, not single-serve: the systemd `opencode-serve`
(`:4096`) **plus ~14 standalone `opencode` TUIs**, each embedding its own server
and each holding `~/.local/share/opencode/opencode.db` open (~15 writers). The
cutover must stop **all** of them — a single stray 1.15 (or 1.17) process on the
shared DB during the switch reintroduces split-brain.

---

## Phase 0 — Cut a fork 1.17 release (carries our patches)

Patches don't touch the DB layer, so DB behavior == upstream v1.17.2.

1. Rebase a `release/v1.17` branch on `johnnymo87/opencode-patched` from upstream
   `v1.17.2`, re-applying our patches (`retry-cap.patch`, `caching.patch`, vim).
   Verify each applies; resolve conflicts.
2. Build the release:
   ```bash
   gh workflow run build-release.yml -R johnnymo87/opencode-patched \
     --ref release/v1.17 --field version=1.17.2 --field revision=1
   ```
3. After it publishes, prefetch hashes for each of the 4 platform assets:
   ```bash
   nix store prefetch-file --json \
     https://github.com/johnnymo87/opencode-patched/releases/download/v1.17.2-patched.1/<asset> \
     | jq -r .hash
   ```

## Phase 1 — Stage the pin (do NOT switch yet)

4. In `users/dev/home.base.nix`, the `opencode` derivation:
   ```nix
   upstreamVersion = "1.17.2";
   patchedRevision = "1";   # -> v1.17.2-patched.1
   ```
   and update the 4 `opencode-platforms` hashes. Update the
   `# V2 DB CORRUPTION HOLD` comment to "cutover in progress".
5. Commit on a branch. Don't `switch` yet.

## Phase 2 — Pre-cutover (pick a DB strategy)

6. **Quiesce.** Stop spawning new sessions; do it during the nightly reset window
   or a manual reset (`resetting-workspace` skill).
7. **Back up** (online snapshot, safe against the live serve):
   ```bash
   python3 -c 'import sqlite3,os,datetime;s=sqlite3.connect(os.path.expanduser("~/.local/share/opencode/opencode.db"));d=sqlite3.connect(os.path.expanduser("~/.local/share/opencode/opencode.db.bak-"+datetime.date.today().isoformat()+"-precutover"));s.backup(d);s.close();d.close()'
   ```
8. **Choose the path:**

   **Fresh + archive (recommended).** Clean, fast, smallest switch-time risk.
   History stays searchable via the archived file. Old sessions are searchable
   but not live-resumable. (Optionally add an `oc-search --db <path>` flag to
   point at the archive.)

   **Migrate in place.** Keeps all sessions live-resumable (`opencode -s <id>`),
   no `oc-search` change — but carries the 4.2 GB bloat + hybrid-DB cruft.
   **Requires** the seq fix-up (replays the migration our repair undid; without
   it 1.17 cannot write). Run with the serve **stopped** (Phase 3 step 9 first):
   ```sql
   DELETE FROM session_message;            -- 959 stale, re-derivable projection rows
   DROP INDEX IF EXISTS session_message_session_seq_idx;
   DROP INDEX IF EXISTS session_message_session_type_seq_idx;
   ALTER TABLE session_message ADD COLUMN seq integer NOT NULL;
   CREATE UNIQUE INDEX session_message_session_seq_idx ON session_message (session_id, seq);
   CREATE INDEX session_message_session_type_seq_idx ON session_message (session_id, type, seq);
   ```
   (Only needed because our DB was repaired. A never-repaired DB skips this.)

## Phase 3 — Atomic switch (no mixed binaries on the DB)

9. **Stop ALL opencode processes** (the multi-writer set):
   ```bash
   sudo systemctl stop opencode-serve.service
   pkill -f '\.opencode-wrapped'   # the ~14 standalone TUIs; verify none hold the DB
   # confirm: no process has opencode.db open (check /proc/*/fd)
   ```
10. **(Migrate path)** apply the seq fix-up SQL from step 8 now.
    **(Fresh path)** move the DB aside:
    ```bash
    cd ~/.local/share/opencode && for f in opencode.db opencode.db-wal opencode.db-shm; do
      [ -e "$f" ] && mv "$f" "$f.archive-$(date +%Y%m%d)"; done
    ```
11. Apply the new profile (host-correct target):
    ```bash
    nix run home-manager -- switch --flake .#cloudbox   # devbox: .#dev
    readlink -f ~/.nix-profile/bin/opencode             # ...-1.17.2-patched.1
    ```
12. Start serve: `sudo systemctl start opencode-serve.service` (fresh path
    creates an empty DB; migrate path opens the fixed-up DB).

## Phase 4 — Verify

13. `curl -sf http://127.0.0.1:4096/global/health` → `version":"1.17.2"`.
14. `opencode-launch` a trivial session → it gets ≥1 message (not blank).
15. Dispatch a subagent / Task → it projects. Orphan check should be 0 for NEW
    subagents (filter by recent `time_created`; old subagents are historically
    orphaned and irrelevant):
    ```sql
    SELECT count(*) FROM session
    WHERE parent_id IS NOT NULL AND time_created > <cutoff_ms>
      AND id NOT IN (SELECT aggregate_id FROM event_sequence);
    ```
16. **(Migrate path)** open a few old sessions (`opencode -s <id>`) — confirm load.
17. **Self-compact regression (known 1.16 fast-follow):** in an interactive
    session, trigger `self_compact_and_resume`; confirm the resumption turn
    actually fires. If it stalls (message sits, nothing happens), that's the open
    fast-follow — file it; it does not corrupt the DB.

## Phase 5 — Guardrails

18. Gate the `update-opencode-patched` automation so it tracks the 1.17 line and
    won't auto-bump unexpectedly.
19. Keep the `precutover` backup for ~2 weeks. Remove the `V2 DB CORRUPTION HOLD`
    comment from `home.base.nix` once stable.

## Rollback

```bash
sudo systemctl stop opencode-serve.service
pkill -f '\.opencode-wrapped'
cp ~/.local/share/opencode/opencode.db.bak-<date>-precutover ~/.local/share/opencode/opencode.db
# revert home.base.nix pin to 1.15.13 / patchedRevision 3, then:
nix run home-manager -- switch --flake .#cloudbox
sudo systemctl start opencode-serve.service
```
