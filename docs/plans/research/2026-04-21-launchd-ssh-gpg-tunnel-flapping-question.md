# Persistent launchd SSH tunnel for GPG agent forwarding flaps constantly on macOS — `IgnoreUnknown=UseKeychain` workaround appears ineffective; what is the correct architecture?

**Keywords:** macOS, launchd, OpenSSH, GPG agent forwarding, RemoteForward, UNIX domain socket, `IgnoreUnknown`, `UseKeychain`, `StreamLocalBindUnlink`, `ExitOnForwardFailure`, nix-darwin, home-manager, autossh, mosh

## The situation

We run a macOS workstation (nix-darwin + home-manager) that connects to two Linux dev boxes:

- **devbox** — Hetzner NixOS VM
- **cloudbox** — GCP NixOS ARM VM

We do all interactive work on the Linux boxes (mosh + tmux + nvim + opencode CLI), but our GPG signing key only lives in the local macOS keyring (Touch ID via 1Password is the pinentry). To allow the Linux boxes to sign git commits without the private key being present there, we forward our local macOS gpg-agent's `extra` socket to the Linux boxes via SSH `RemoteForward` of a UNIX domain socket:

```
RemoteForward /run/user/1000/gnupg/S.gpg-agent /Users/<user>/.gnupg/S.gpg-agent.extra
```

To keep this forward alive 24/7 (so headless / non-interactive opencode sessions can sign), we run a dedicated launchd LaunchAgent on macOS that runs `ssh -N <gpg-tunnel-host>` with `KeepAlive=true`. We have one agent per remote: `devbox-gpg-tunnel` and `cloudbox-gpg-tunnel`.

Today we discovered that GPG signing on both Linux boxes is broken. After investigation we found:

1. The forwarded socket is missing or stale on the remote.
2. The launchd agent has restarted **dozens of times today** (`runs = 33`).
3. The error logs are dominated by an SSH config parse error that means the SSH process **never even attempts the connection** — it dies during config parse.

Manually kickstarting the agent + clearing stale remote sockets restores signing immediately, but the tunnel will flap again on its own.

## Environment

- macOS (Apple Silicon), OpenSSH client `10.2p1` from nixpkgs (`/nix/store/waxzdvhz2hmxpkjgjdp558q0bc67l94k-openssh-10.2p1/bin/ssh`)
- Configuration via nix-darwin + standalone home-manager
- LaunchAgent generated by home-manager from `home.darwin.nix`
- Both Linux remotes run NixOS with OpenSSH server, `ClientAliveInterval 30 / ClientAliveCountMax 3`, `StreamLocalBindUnlink yes`
- macOS `pmset` shows `sleep 0 (sleep prevented by powerd)`, `tcpkeepalive 1`, `powernap 1`. The Mac is awake / power-connected. `displaysleep 10`, `disksleep 10`.
- The Mac has many network interfaces (Wi-Fi, two Thunderbolt Ethernet adapters, several `utun*` VPN interfaces). We do switch between Wi-Fi and Thunderbolt Ethernet over the course of a day.
- Mosh is used for interactive sessions; mosh uses UDP and is unaffected by these tunnel issues.

## Relevant `~/.ssh/config` (generated by `scripts/update-ssh-config.sh`)

The very top of `~/.ssh/config` (line 5 specifically) sets `UseKeychain yes` under `Host *`:

```
Host *
    UseKeychain yes        # <-- line 5 — Apple-only ssh_config option
    AddKeysToAgent yes
    IdentityAgent ~/Library/Group Containers/2BUA8C4S2C.com.1password/t/agent.sock
    ...

Host devbox-gpg-tunnel
    HostName <devbox public IP>
    User dev
    ServerAliveInterval 60
    ServerAliveCountMax 3
    StreamLocalBindUnlink yes
    RemoteForward /run/user/1000/gnupg/S.gpg-agent /Users/<user>/.gnupg/S.gpg-agent.extra

Host cloudbox-gpg-tunnel
    HostName <cloudbox public IP>
    User dev
    ServerAliveInterval 60
    ServerAliveCountMax 3
    StreamLocalBindUnlink yes
    RemoteForward /run/user/1000/gnupg/S.gpg-agent /Users/<user>/.gnupg/S.gpg-agent.extra
```

`UseKeychain` is an Apple-patched OpenSSH option that the **nixpkgs OpenSSH 10.2p1 binary does not recognize**. The home-manager author tried to handle this by passing `-o IgnoreUnknown=UseKeychain` on the command line.

## What the err logs reveal (sorted by frequency, last week)

For `cloudbox-gpg-tunnel.err.log` (7138 lines total):

```
2977  /Users/.../.ssh/config: terminating, 1 bad configuration options
2977  /Users/.../.ssh/config: line 5: Bad configuration option: usekeychain
 417  client_loop: send disconnect: Broken pipe
 336  Read from remote host <ip>: Connection reset by peer
 193  ssh: connect to host <ip> port 22: Network is unreachable
  73  connect to /Users/.../.gnupg/S.gpg-agent.extra port -2 failed: No such file or directory
  29  Read from remote host <ip>: Operation timed out
```

For `devbox-gpg-tunnel.err.log` (6979 lines total) the distribution is essentially identical.

**The dominant error — by ~7x over the next most common — is the `UseKeychain` parse error.** This means the SSH process **dies during config parsing and never reaches the network**. So `IgnoreUnknown=UseKeychain` is not having the desired effect in this configuration.

Since `KeepAlive=true` plus `ThrottleInterval=120` is on the launchd job, when ssh exits during config parsing, launchd waits 120s and respawns. That gives a ~720 restarts/day theoretical max. We observed 33+ restarts in one workday.

**However**, sometimes the tunnel does succeed — we have observed the forwarded socket appearing on the remote. So the parse error is intermittent, not 100% — which is itself confusing because the ssh_config file is static.

## Hypotheses (clearly marked as guesses)

1. **`IgnoreUnknown=UseKeychain` from the command line is processed *after* the ssh_config file is parsed** in this OpenSSH version, so by the time `IgnoreUnknown` would take effect, ssh has already aborted on `UseKeychain`. We have not verified this against OpenSSH source for 10.2p1; this is a guess based on observed behavior. But this would explain ~all of the parse-error log lines.

2. **The intermittent successes might be because something else is occasionally invoking these LaunchAgents in a different working directory, or with a different `~/.ssh/config` fingerprint.** Or some race in how the file is being rewritten by `scripts/update-ssh-config.sh` (which dynamically discovers the GCP IP). Unverified.

3. **Even when the tunnel does come up, the forwarded socket can become "stale" on the Linux side** — present as a file but not routing to a live agent. We've verified that running `rm -f /run/user/1000/gnupg/S.gpg-agent` on the remote, then `launchctl kickstart -k <agent>`, restores signing. The remote's sshd has `StreamLocalBindUnlink yes` and the SSH client config also has `StreamLocalBindUnlink yes`, but apparently this is not always cleaning up.

4. **`ExitOnForwardFailure=yes`** combined with the above means even minor bind hiccups kill the connection and trigger a 120s respawn cycle.

5. **The `-o port -2 failed` error**: `port -2` is OpenSSH's sentinel for a UNIX domain socket forward. The "No such file or directory" refers to the **local** Mac socket `~/.gnupg/S.gpg-agent.extra` — meaning at the moment of forwarding, the Mac's gpg-agent had not yet created its `.extra` socket. This happens occasionally after macOS resumes from sleep or when gpg-agent has been killed for unrelated reasons.

## What we know for certain (verified this session)

- Manual `ssh cloudbox-gpg-tunnel "echo OK"` from a normal shell works fine — connection succeeds, RemoteForward establishes the socket on the remote, signing works. Same for devbox.
- `launchctl kickstart -k gui/$(id -u)/org.nix-community.home.cloudbox-gpg-tunnel` does cause the agent to launch and the socket to appear on the remote — sometimes signing then works, sometimes the socket exists but is dead and we need to `rm -f /run/user/1000/gnupg/S.gpg-agent` on the remote and kickstart again.
- Manual ssh works because it's run from an interactive shell, where ssh sees `~/.ssh/config` the same way the launchd ssh does — but somehow no `UseKeychain` parse error appears interactively. **We have not understood why interactive ssh tolerates `UseKeychain` while launchd-spawned ssh doesn't.** Both are the same `/nix/store/.../openssh-10.2p1/bin/ssh` binary. (This is one of the questions we want answered.)
- Local Mac gpg-agent is healthy: `gpg --list-secret-keys` returns the key, signing locally works.
- `ServerAliveInterval=30 / Count=3` should drop dead connections in ~90s. Server-side `ClientAliveInterval=30 / Count=3` also enforces this from the other side.

## Specific questions we want answered

1. **Why does `-o IgnoreUnknown=UseKeychain` on the command line appear to be ineffective when launchd spawns ssh, but `Host *  UseKeychain yes` in ssh_config doesn't error when ssh is run from an interactive shell?** Is this an OpenSSH 10.2p1 quirk, a launchd-environment quirk (e.g., different `$HOME`, missing TERM, missing PATH), or something about the order in which OpenSSH applies `-o` flags relative to ssh_config?

2. **What is the canonical, reliable way to run a long-lived SSH UNIX-socket forwarding tunnel on macOS with launchd?** We've found articles suggesting `autossh` (which seems abandoned), various wrappers, and `KeepAlive=true` with bare ssh. Is there a 2025/2026 best practice — particularly one that survives:
   - Wi-Fi ↔ Thunderbolt Ethernet network interface changes
   - macOS sleep/wake (Power Nap on, sleep prevented)
   - The remote sshd evicting idle clients (`ClientAliveInterval=30 Count=3`)
   - VPN bring-up/teardown (we have several `utun*` interfaces, occasionally use Twingate / WARP)

3. **What is the right way to handle the "stale UNIX domain socket on remote" problem for `RemoteForward /path /local-path`?** We have `StreamLocalBindUnlink yes` set on both ends — what conditions defeat it? Is there a way to atomically detect & recover when the socket file exists but doesn't route to a live agent on the macOS side?

4. **For the `RemoteForward` of a gpg-agent socket specifically**: are there well-known patterns that handle the "Mac's gpg-agent restarted while the tunnel was up" case (which leaves a forwarded socket pointing nowhere)? In particular — is there a way to make ssh re-establish the forward when the local end goes away, without tearing down the entire SSH connection?

5. **Should we abandon `RemoteForward` of the gpg-agent socket entirely and use a different architecture?** Options we're aware of (please critique):
   - `gpg-agent --remote-only` mode running on the remote with restricted scope
   - Using `1password-cli` (`op`) on the remote, with `op signin` and the GPG private key stored in a 1Password item instead of locally
   - A YubiKey + smartcard forwarding (`ssh -o RemoteForward=...` of `S.scdaemon`)
   - SSH commit signing instead of GPG (git supports `gpg.format = ssh` since 2.34) — keeping the SSH key in 1Password's SSH agent
   - Storing the GPG private key on the Linux box itself (we explicitly do not want this, since it would defeat the Touch-ID-required-to-sign property)

6. **Specifically for the home-manager + nix-darwin + nixpkgs OpenSSH situation**: is there a known-good way to make Apple-patched OpenSSH options coexist with vanilla OpenSSH in `~/.ssh/config`? Either:
   - Make all binaries tolerant of `UseKeychain`, or
   - Conditionally include `UseKeychain` only when the binary supports it, or
   - Use the system Apple OpenSSH for some hosts and nixpkgs OpenSSH for others, or
   - Drop `UseKeychain` entirely (we use 1Password's SSH agent via `IdentityAgent`, so `UseKeychain` may already be redundant)

## Constraints

- The GPG private key MUST stay on macOS (Touch ID protection). Importing it onto the Linux boxes is not acceptable.
- Configuration must be expressible in nix-darwin / home-manager (no hand-edited plists or post-install scripts that defeat reproducibility).
- We can install additional packages from nixpkgs.
- We can change `~/.ssh/config` generation (it's already a script we wrote — `scripts/update-ssh-config.sh`).
- We use `KeepAlive = true` and `ThrottleInterval = 120` in launchd; we are open to changing these.
- We need this to work for headless / non-interactive opencode sessions (so we can't rely on GPG agent forwarding being established by an interactive ssh session).

## What we'd like in the answer

A reasoned recommendation between (a) fixing the current architecture (launchd-managed `ssh -N` + RemoteForward UNIX socket), (b) replacing it with a different forwarding approach, or (c) replacing GPG signing with something else (SSH signing, etc.). We'd value an explanation of *why* each option is robust or fragile, not just a config snippet.

If the answer is "fix the current architecture," we'd like a worked-through fix for the `UseKeychain` parse issue and concrete advice on stale-socket recovery and network-change resilience.

If the answer is "switch to SSH commit signing," we'd like to know the specific ergonomic / verification trade-offs vs. GPG (e.g., GitHub displays "Verified" for both, but local `git log --show-signature` behaves differently; how does that interact with 1Password's SSH agent and Touch ID?).
